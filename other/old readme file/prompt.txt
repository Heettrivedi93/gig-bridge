## 💰 Payment Flow (Buyer → Seller)

### Step-by-step

1. **Buyer places an order** — selects a gig package, fills requirements, and proceeds to payment
2. **Payment is initiated** — buyer is redirected to PayPal to complete payment
3. **Payment is captured** — amount is charged from buyer and held by the platform (escrow-like)
4. **Order becomes Active** — seller is notified to start working
5. **Seller delivers work** — uploads delivery file and note
6. **Buyer reviews delivery** — accepts or requests revision
7. **Order marked Completed** — funds are released to seller's account
8. **Seller can withdraw** — earnings become available for payout

### Payment States

| State | Description |
|-------|-------------|
| pending | Payment initiated but not confirmed |
| paid | Payment successfully captured |
| failed | Payment attempt failed |
| refunded | Payment refunded to buyer (on cancellation) |
| released | Funds released to seller after order completion |

### Platform Fee Logic

- Platform deducts a **service fee** (e.g. 10–20%) from the order amount
- Remaining amount is credited to the seller
- Example: Buyer pays **$100** → Platform fee **$10** → Seller receives **$90**

### Cancellation & Refund

- If order is cancelled **before** seller starts → full refund to buyer
- If order is cancelled **after** work started → partial refund based on dispute resolution
- Refunds are processed back via PayPal

### Key Database Fields Involved

| Table | Relevant Fields |
|-------|-----------------|
| orders | buyer_id, seller_id, gig_id, package_id, price, status, payment_status, escrow_held |
| payments | order_id, user_id, method, amount, status, transaction_id, paypal_order_id, paypal_payer_id |
| subscriptions | user_id, plan_id, starts_at, ends_at, status |
